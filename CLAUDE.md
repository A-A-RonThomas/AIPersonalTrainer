# AI Trainer — Project Context

A weekly workout-programming service that acts as a personal trainer. The user defines a goal; the service generates a week of training on a Push/Pull/Legs(+core) skeleton, the user completes the workouts in Hevy, and the next week is generated by evaluating the prior weeks' actual performance. Run cadence is **once per week**, at the week boundary — not real-time.

Currently single-user (me). No onboarding flow, no broad exercise library, no beginner guardrails needed. Working loads, equipment, and split are seeded from my own Hevy history and hardcoded.

## Core concept (one line)

A weekly loop that holds the PPL skeleton fixed, scores last week's per-exercise history against the current energy-balance goal, and progresses / holds / backs off each lift accordingly — with freeform notes as the override signal.

## The two goals

There are effectively two goals: **build muscle** and **preserve muscle (during weight loss)**. Modifiers like "as fast as safely possible" and "avoid injury" are NOT separate goals — they're a single autoregulation dial (aggressive = bigger load jumps, slow to back off; conservative = smaller jumps, quick to deload on a pain note).

My current goal: **muscle preservation during a cut.**

### Generation barely differs between the two goals
Training stimulus is the same for build vs. preserve — energy balance (handled via diet, outside this tool) is what decides gain vs. loss. Both use the same PPL skeleton, heavy compounds, and rep ranges. **Keep loads heavy in a cut** — dropping to light weight is the classic muscle-losing mistake. The one real generation difference: run **leaner total volume in a deficit** (worse recovery; junk volume only digs a recovery hole).

### Evaluation FORKS hard on goal
The same data means opposite things depending on goal. "Load flat for 3 weeks" is a **stall needing intervention** when bulking, but a **win** when cutting (strength preserved while losing fat). The eval engine must know the current energy-balance context or it will misread a successful cut as failure.

### Goal is flippable STATE, not a fixed setting
Goal can change between weeks (cut → maintenance → lean bulk). When it flips, evaluation recalibrates its expectations. Nothing gets rebuilt.

## Architecture: deterministic engine + Claude layer

Keep the numbers deterministic. Use Claude for the fuzzy, qualitative parts. Do NOT put load progression inside the LLM — it must be consistent and trustworthy.

**Deterministic (plain code):**
- goal → set/rep/intensity/rest scheme (lookup table)
- training days/week → split layout across the 7 days
- rep math: "hit all prescribed reps → progress load"
- stall detection over rolling history → trigger deload / rep-range change
- guardrails: hard caps on load jumps so a bad note-parse can never propose e.g. a +50 lb jump

**Claude:**
- exercise selection & rotation (variability while keeping muscle coverage constant)
- reading freeform notes ("left shoulder twinged on incline", "bored of barbell rows", "last set brutal") — this is the qualitative signal that turns a generator into a coach
- the trainer-voice explanation of *why* each change was made
- handling the chat-based plan edits

Notes are **optional** and must degrade gracefully: with no notes, eval runs on pure completion data.

## Per-exercise weekly evaluation

Each week, per movement, land on one decision: **progress load** (hit reps cleanly / note says easy), **hold** (hit reps but barely), **repeat or deload** (missed reps), or **swap** (note flags pain or boredom). Rep math is deterministic; interpreting notes and choosing the swap is the Claude part.

## Rep ranges (reference)
- Hypertrophy / muscle: 6–12 reps, RPE 7–9, 60–120s rest, isolation accessories, progress reps then load.
- Strength emphasis: 3–5 reps, heavy loads, 3–5 min rest, compounds prioritized.

## Data model

The plan is **structured state**, not a transcript. Chat edits operate on a plan JSON object; Claude returns a revised plan object plus a short human explanation of what changed. The UI renders the JSON; the prose goes in the chat log. "Approve" commits a specific object.

History must be **rolling per-exercise**, not just last week — enough depth to distinguish steady progress from a stall (e.g. "load hasn't moved in 3 weeks → deload or change rep range"). Persist program state on the home server (Postgres, alongside FamilyBudget).

**Cold start:** Week 1 has no history. Needs a cold-start mode (goal + working loads seeded from Hevy). Treat the first couple of weeks as calibration (finding real working loads), not real progression.

## PPL skeleton

Days are defined by movement-pattern muscle-coverage SLOTS, not fixed exercises — slots are the invariant that guarantees the same muscles get hit. Variability comes from rotating which exercise fills each slot, biased away from recent picks. Optionally pin an anchor compound for progression and rotate only accessories.

- **Push:** chest, shoulders, triceps
- **Pull:** back/lats, rear delts, biceps (slots e.g. vertical pull, horizontal pull, rear delt, biceps, grip)
- **Legs/core:** quads, hamstrings, glutes, calves, core

Days/week drives the split — **do not hardcode 6-day PPL.** Pure PPL is 3 or 6 days; 4–5 days needs a hybrid (PPL + upper/lower, or PPL with a rest day inserted). Take training days as input.

## Hevy integration

- Hevy API requires a **Hevy Pro** account; key from hevy.com/settings?developer.
- Hevy explicitly warns the API may change or be abandoned — wrap it in a thin adapter layer that can be swapped.
- Relevant endpoints: `GET /v1/workouts` (completed sessions), `GET /v1/exercise_history/{id}`, `GET /v1/exercise_templates` (each carries primary/secondary muscle tags — needed to map slots → real exercises), `GET /v1/routines`, `POST /v1/routines` (create), `PUT /v1/routines/{id}` (update). IDs are UUIDs.
- Exercise selection must resolve to real exercise_template IDs — never freeform exercise names that won't map.

## Front end & review gate

Single chat-driven review surface (the review screen IS the conversation). Flow: generate week → render → approve as-is OR chat edits ("swap the barbell rows, shoulder's cranky" / "bump squat volume" / "only 4 days this week") → regenerate affected pieces → approve → **then** push to Hevy.

**Hard rule: nothing writes to Hevy until explicit approval.** Chat edits the plan object, NOT Hevy directly — the LLM never touches the training log, it drafts and the user commits. The approval gate is also the safety net against a bad note-parse.

Stack: FastAPI + Jinja2 on the home server behind Tailscale (same pattern as FamilyBudget). Render the week as cards (day → exercises → sets/reps/load) with a **diff vs last week** so changes are visible. Chat box posts edit requests; backend round-trips the Anthropic API, re-renders, Approve button fires Hevy writes. HTMX optional for live-feel chat without full reloads; plain forms are fine for a weekly single-user tool.

## Scope boundary

It programs and progresses training. It does **not** coach form or manage nutrition — those are separate, harder problems left out of scope.
